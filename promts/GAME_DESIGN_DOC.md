# NFT-DND: Procedural Real-Time Multiplayer RPG

## Technical and Game Design Document (GDD & TDD)

## Introduction

**NFT-DND** is a procedural, real-time multiplayer RPG with a browser client,
where key gameplay elements such as lore, quest structure, NPC parameters, and
loot are dynamically generated by an AI Dungeon Master. The project is now
designed around **OneChain** for on-chain legendary items, achievement history,
and major world events.

## 1. AI Dungeon Master Architecture

The central game brain operates as a distributed AI pipeline that analyzes the
world state and generates content.

### 1.1 Core components

| Component | Purpose | Suggested implementation |
| :--- | :--- | :--- |
| **LLM core** | Generates dialogue branches, lore, and quest descriptions. | Direct or proxy integration with OpenAI or Anthropic using strict system prompts. |
| **Narrative director** | Manages session context, pacing, and story continuity for each party. | Vector database memory with retrieval-augmented generation. |
| **Adaptive difficulty engine** | Adjusts encounter difficulty in real time. | Rule-based balancing using DPS, win rate, and group gear score. |
| **World state tracker** | Tracks coordinates, triggers, and NPC state. | Redis for real-time state plus PostgreSQL for persistence. |
| **Player behavior analyzer** | Measures play style such as aggression, stealth, diplomacy, and greed. | Action scoring and periodic profile updates. |
| **Loot and lore minting service** | Creates unique legendary loot and narrative metadata. | Metadata generation plus IPFS storage and OneChain Move calls. |

### 1.2 AI behavior logic

- The AI consumes a typed state payload and returns deterministic JSON updates.
- A pre-generated story skeleton exists, but the AI can restructure scenes when
  players break expected quest flow.
- Inputs include party HP and mana, inventory, recent dialogue choices, time in
  location, and player wallet or on-chain identifiers when needed.
- If a party stalls, the system can trigger hints, cave-ins, patrols, or
  traveling merchants to avoid dead ends.
- Procedural templates constrain the majority of content generation, while a
  smaller portion is reserved for creative AI output.

## 2. World system

The world is a minimalist top-down 2D environment rendered in the browser while
remaining server-authoritative.

### 2.1 Procedural generation

- The world uses a tile map or hex grid.
- Zones are defined by biome, threat level, resource density, and magical
  modifiers.
- New subzones can be expanded on demand when player density increases.

### 2.2 Reusable templates

- Room types: arena, puzzle room, treasure room, corridor, safe zone
- Mob types: swarm, tank, caster, boss
- Dungeon pathing: entrance, combat rooms, branch, trap or puzzle, boss room
- Event chains: rescue, betrayal, hidden battle; key hunt, puzzle, reward

## 3. Multiplayer layer

- WebSockets broadcast movement, combat, narrative, voting, and spawn events.
- Parties are designed for 1 to 5 players.
- Cooperative puzzles are generated around group stat checks and role synergy.
- Hub zones can scale through instancing when concurrency grows.

## 4. OneChain integration

OneChain replaces the previous Stellar-based direction.

### 4.1 Loot as on-chain assets

- Rare and legendary items can be wrapped into metadata objects.
- Metadata can reference IPFS assets and AI-generated backstory.
- Publishing and transfer are intended to happen through Move packages deployed
  with the `one` CLI.

### 4.2 Chronicle history

Only major milestones should be recorded on-chain:

- first kills of major bosses
- server-wide world decisions
- seasonal achievement badges

### 4.3 Contract direction

The repository now contains a `Move` package that can evolve into:

- legendary item minting
- chronicle badge issuance
- event-based world history recording

## 5. Progression and loot

- Core attributes remain strength, agility, intelligence, charisma, and luck.
- The build system is classless and skill-tree driven.
- Rare items receive AI-generated lore tied to how they were found.
- Synergy between party members remains central to combat design.

## 6. Interface

The UI should remain minimal, readable, and atmospheric:

- central game canvas
- side narrative log for AI Dungeon Master output
- modal quest or voting overlays
- lightweight HUD for stats and party coordination
